## ARCHITECTURE & SECURITY

### 1. Core Distinctions
* **Authentication (AuthN):** Proving *who* you are. (e.g., Email/Password login, validating a JWT). 
* **Authorization (AuthZ):** Proving *what* you can do. (e.g., RBAC checking if the authenticated user has the `"ADMIN"` role to delete a record).
* **User vs. Session:**
  * **User:** The core entity (`users` table). Represents the human. 
  * **Session:** A specific device or client’s connection to that user (`user_sessions` table). 
  * *Rule:* 1 User = N Sessions. (MacBook and iPhone are separate sessions).

### 2. Security Fundamentals
* **Why hash passwords?** If a database is compromised, plain text passwords allow attackers to hijack accounts across the internet due to password reuse. Hashing (`bcrypt`) is a mathematical one-way street.
* **Why JWT (JSON Web Tokens)?** Access tokens are cryptographically signed. The server validates the signature mathematically without hitting the database on every API request, drastically reducing DB load and latency.

### 3. JWT Architecture (Access vs. Refresh)
This solves the flaw of standard JWTs: *you cannot easily revoke a stateless token before it expires.*
* **Access Token (Short-lived):** Lives for ~15 minutes. Contains the user's `role` and identity. Sent with every API request.
* **Refresh Token (Long-lived):** Lives for 7+ days. Only sent to the `/auth/refresh` endpoint to acquire a fresh Access Token. Contains only `user_id` and `session_id`.

### 4. Session Management & Control
* **Why a `user_sessions` table?** It gives stateful control over stateless JWTs. By embedding a `session_id` inside the JWT payload, we can remotely track, manage, and kill specific device logins.
* **How Logout Works:** Client hits `POST /auth/logout`. Server reads `session_id` from the token and sets `is_revoked = True` in the `user_sessions` table. When the Access Token expires, the Refresh Token is denied.
* **Token Rotation:** When a Refresh Token is used, the server issues a *new* Access Token and a *new* Refresh Token, invalidating the old one. If an old token is reused, the server detects theft and revokes the entire session cascade.

## TECHNICAL REFERENCE & COMMANDS

### 1. Git (Version Control)
**Key Terminology:**
* **Commit:** A saved snapshot of your code.
* **Branch:** An independent line of development.
* **Merge:** Combining changes from one branch into another.
* **Stash:** Temporarily shelving uncommitted changes.

**Major Commands:**
```bash
git init                                 # Initialize a new repository
git clone <url>                          # Clone an existing repository
git status                               # Check the status of your files
git checkout -b <branch_name>            # Create and switch to a new branch
git add .                                # Stage all changed files
git commit -m "feat: add user auth"      # Create a commit
git push origin <branch_name>            # Push commits to remote
git pull origin <branch_name>            # Fetch and merge changes
git log --oneline                        # View commit history
git reset --soft HEAD~1                  # Undo last commit but keep changes in editor
```

### 2. Docker & Docker Compose (Infrastructure)
**Key Terminology:**
* **Image:** A read-only template (the blueprint).
* **Container:** A runnable instance of an image.
* **Volume:** A mechanism for persisting data generated by containers (crucial for databases).
* **Port Mapping:** Connecting a port on your host machine to a port inside the container (`host_port:container_port`).

**Major Commands:**
```bash
docker build -t my-app:latest .          # Build an image
docker run -d -p 8000:8000 my-app:latest # Run container in detached mode with port mapping
docker ps                                # List running containers
docker ps -a                             # List ALL containers
docker logs -f <container_name>          # View live logs
docker exec -it <container_name> /bin/bash # Open shell inside running container
docker-compose up --build -d             # Build and start all services in background
docker-compose down -v                   # Stop containers and destroy volumes (WARNING: Deletes DB data)
```

### 3. PostgreSQL (Database)
**Key Terminology:**
* **Schema:** Logical blueprint of how data is organized.
* **Primary Key:** Unique identifier for each record.
* **Foreign Key:** Field that links to another table (creates relationships).
* **Index:** Data structure that improves data retrieval speed.

**Major Commands (psql):**
```bash
psql -U postgres                         # Access terminal
\c my_database                           # Connect to DB
\l                                       # List all databases
\dt                                      # List tables
\d table_name                            # View table schema
\q                                       # Quit
```

### 4. SQLAlchemy (Python ORM)
**Key Terminology:**
* **Engine:** Starting point; manages connection pools.
* **Session:** Establishes conversations with the database.
* **Model:** A Python class mapping to a specific DB table.

**Basic Setup:**
```python
from sqlalchemy import create_engine, Column, Integer, String
from sqlalchemy.orm import declarative_base, sessionmaker

engine = create_engine("postgresql://user:password@localhost/dbname")
SessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine)
Base = declarative_base()

class User(Base):
    __tablename__ = "users"
    id = Column(Integer, primary_key=True, index=True)
```

### 5. Alembic (Database Migrations)
**Key Terminology:**
* **Migration/Revision:** Script applying changes to a DB schema.
* **Head:** The most recent migration revision.
* **Upgrade/Downgrade:** Moving the DB schema forward or backward in time.

**Major Commands:**
```bash
alembic init alembic                     # Initialize Alembic
alembic revision --autogenerate -m "msg" # Generate a migration script
alembic upgrade head                     # Apply pending migrations
alembic downgrade -1                     # Revert the last migration
```

### 6. FastAPI (Web Framework)
**Key Terminology:**
* **Path Operation:** URL path + HTTP method (e.g., `POST /users`).
* **Dependency Injection:** Framework automatically provides required code (e.g., passing DB session to route).
* **Pydantic Model:** Validates data entering and leaving the API.

**Major Commands & Usage:**
```bash
uvicorn main:app --reload                # Start dev server
```
```python
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

class ItemCreate(BaseModel):
    name: str

@app.post("/items/", status_code=201)
def create_item(item: ItemCreate):
    return item
```

### 7. Redis (Caching & Sessions)
**Key Terminology:**
* **Caching:** Store heavy DB query results for instant retrieval.
* **Rate Limiting:** Track API hits per IP in a time window.

**Major Commands (redis-cli):**
```bash
redis-cli                                # Open Redis terminal
PING                                     # Check if alive (returns PONG)
SET mykey "value" EX 3600                # Set key with 1-hour expiration
GET mykey                                # Retrieve key
FLUSHALL                                 # Wipe entire Redis database
```
